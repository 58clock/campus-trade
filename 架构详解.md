**关键设计决策：**

- `map-underscore-to-camel-case=true`：这是整个项目中最容易被忽略但最重要的配置。它让你写 `private Integer loginFailCount` 就能自动映射数据库的 `login_fail_count` 列。如果没有这个配置，每个 Entity 字段都需要 `@TableField("login_fail_count")`。
- `app.jwt.secret`：Base64 编码是 jjwt 0.12.x 的新要求。老版本 jjwt 直接用字符串做密钥（内部自动转），新版本必须显式解码。
- `logic-delete-field: deleted`：当前配置但实际上所有 Entity 都没有 `deleted` 字段，所以逻辑删除功能处于未激活状态。如果将来需要逻辑删除，需要给 Entity 添加 `private Integer deleted;`（类型需匹配配置）。

### 2.4 `CampusTradeApplication.java` — 启动类

**文件路径：** `backend/src/main/java/com/campus/CampusTradeApplication.java`

```java
@SpringBootApplication  // = @Configuration + @EnableAutoConfiguration + @ComponentScan
@MapperScan("com.campus.mapper")  // MyBatis-Plus Mapper 扫描（不是 Spring 的 @Repository）
public class CampusTradeApplication {
    public static void main(String[] args) {
        SpringApplication.run(CampusTradeApplication.class, args);
    }
}
```

**关键点：**
- `@MapperScan("com.campus.mapper")`：这是 MyBatis-Plus 的注解，告诉框架在 `com.campus.mapper` 包下寻找所有标注了 `@Mapper` 的接口。它不能用 `@Repository` 替代，因为 MyBatis-Plus 在运行时为每个 Mapper 接口生成 JDK 动态代理实现类，然后注册到 Spring 容器中。
- `@SpringBootApplication`：组合注解，其中 `@ComponentScan` 默认扫描当前类所在包及其子包。这就是为什么所有类放在 `com.campus` 包下就能被自动扫描到。

### 3.1 `SecurityConfig.java` — 安全架构核心

**文件路径：** `backend/src/main/java/com/campus/config/SecurityConfig.java`

```java
@Configuration
@EnableWebSecurity       // 启用 Spring Security 的 Web 安全过滤器链
@EnableMethodSecurity    // 启用 @PreAuthorize 方法级注解（在 AdminController 中使用）
@RequiredArgsConstructor // Lombok：为 final 字段生成构造函数
public class SecurityConfig {

    private final JwtTokenFilter jwtTokenFilter;  // 构造函数注入

    // === Bean 1: 安全过滤器链（策略模式配置）===
    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            // 1. 关闭 CSRF 保护
            //    原因：用 JWT 无状态认证，不依赖 Cookie/Session
            //    CSRF 攻击的前提是浏览器自动携带 Cookie，JWT 由 JS 手动添加 Header
            //    所以不存在 CSRF 攻击面
            .csrf(AbstractHttpConfigurer::disable)

            // 2. 无状态会话管理
            //    STATELESS：不创建 HttpSession，每个请求完全独立
            //    Spring Security 不会用 JSESSIONID 追踪用户
            //    认证状态完全由 JWT Token 携带
            .sessionManagement(sm ->
                sm.sessionCreationPolicy(SessionCreationPolicy.STATELESS))

            // 3. URL 权限规则（MATCH ORDER MATTERS！按声明顺序匹配）
            .authorizeHttpRequests(auth -> auth
                // 第一组：完全放行（白名单，不需要任何认证）
                .requestMatchers(
                        "/api/auth/**",        // 注册/登录/忘记密码
                        "/swagger-ui/**",      // API 文档页面资源（CSS/JS）
                        "/swagger-ui.html",    // Swagger 入口
                        "/v3/api-docs/**",     // OpenAPI JSON 数据
                        "/uploads/**"          // 上传的图片文件
                ).permitAll()

                // 第二组：GET 请求公开
                .requestMatchers(HttpMethod.GET,
                        "/api/products",                    // 商品列表
                        "/api/products/{id}",               // 商品详情
                        "/api/products/{id}/messages"       // 商品留言
                ).permitAll()

                // 第三组：管理员独占接口
                // hasRole("ADMIN") 会自动加前缀，实际匹配 ROLE_ADMIN
                .requestMatchers("/api/admin/**").hasRole("ADMIN")

                // 第四组：兜底规则——其他一切都需要认证
                .anyRequest().authenticated()
            )

            // 4. 插入 JWT 过滤器
            //    addFilterBefore：在 UsernamePasswordAuthenticationFilter 之前执行
            //    UsernamePasswordAuthenticationFilter 是表单登录的默认过滤器
            //    我们把它替换成 JWT 解析
            .addFilterBefore(jwtTokenFilter,
                             UsernamePasswordAuthenticationFilter.class);

        return http.build();
    }

    // === Bean 2: 密码编码器 ===
    @Bean
    public PasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder();
        // BCrypt 特点：
        //   1. 自带盐值：每次加密结果不同（盐值嵌入在 hash 前 22 位）
        //   2. 计算强度可调：默认 2^10=1024 轮迭代
        //   3. 不可逆：无法从 hash 反推原文
        //   4. matches(raw, encoded) 能正确比对——盐值从 encoded 中提取
    }

    // === Bean 3: 认证管理器（供手动认证场景使用）===
    @Bean
    public AuthenticationManager authenticationManager(
            AuthenticationConfiguration config) throws Exception {
        return config.getAuthenticationManager();
        // 当前代码中实际上没有使用这个 Bean
        // 但保留它以便将来需要手动调 authenticationManager.authenticate()
    }
}
```

**请求处理流程（完整链路）：**

```
HTTP 请求到达 Tomcat
  │
  ▼
SecurityFilterChain（Spring Security 过滤器链）
  │
  ├─ ① 匹配 permitAll 白名单？
  │     └─ 是 → 跳过认证检查，直接到 Controller
  │
  ├─ ② 匹配 hasRole("ADMIN")？
  │     └─ 是 → JwtTokenFilter 执行（见下面）→ 检查 role == ADMIN → 403 或放行
  │
  └─ ③ 匹配 anyRequest().authenticated()？
        └─ JwtTokenFilter 执行 → 无有效 token → 401
```

**规则顺序为什么重要？** Spring Security 的 `authorizeHttpRequests` 按声明顺序进行 FIRST MATCH。如果 `anyRequest().authenticated()` 写在前面，后续的 permitAll 永远不会被匹配。

### 3.2 `CorsConfig.java` — 跨域配置

**文件路径：** `backend/src/main/java/com/campus/config/CorsConfig.java`

```java
@Configuration
public class CorsConfig implements WebMvcConfigurer {
    @Override
    public void addCorsMappings(CorsRegistry registry) {
        registry.addMapping("/api/**")           // 只对 /api 路径生效
                .allowedOriginPatterns("*")      // 允许任意来源（开发环境用 *，生产需限制）
                .allowedMethods("GET", "POST", "PUT", "DELETE", "OPTIONS")
                .allowedHeaders("*")             // 允许任意请求头
                .allowCredentials(true)          // 允许携带 Cookie/Authorization
                .maxAge(3600);                   // 预检请求 OPTIONS 缓存 1 小时
    }
}
```

**为什么用 `allowedOriginPatterns("*")` 而非 `allowedOrigins("*")`？**
- `allowedOrigins("*")` 和 `allowCredentials(true)` 不能同时使用（浏览器安全规范禁止）
- `allowedOriginPatterns("*")` 是通配符模式，允许配合 credentials
- 严格来说生产环境应该写具体域名，如 `allowedOriginPatterns("https://xxx.com")`

### 3.3 `WebConfig.java` — 静态资源映射

**文件路径：** `backend/src/main/java/com/campus/config/WebConfig.java`

```java
@Configuration
public class WebConfig implements WebMvcConfigurer {

    @Value("${file.upload.path:./uploads}")   // 从配置读取，如未配置则默认为 ./uploads
    private String uploadPath;

    @Override
    public void addResourceHandlers(ResourceHandlerRegistry registry) {
        // Paths.get("./uploads").toAbsolutePath()
        //   → 在 Windows 上变成 D:\project\uploads\
        //   → toUri() → "file:///D:/project/uploads/"
        String absolutePath = Paths.get(uploadPath).toAbsolutePath().toUri().toString();

        registry.addResourceHandler("/uploads/**")     // URL 路径模式
                .addResourceLocations(absolutePath);    // 映射到的文件系统路径
    }
}
```

**这个配置解决什么问题？**

Spring Boot 默认只映射以下位置的静态资源：
- `classpath:/static/`
- `classpath:/public/`
- `classpath:/resources/`
- `classpath:/META-INF/resources/`

用户上传的图片存在文件系统（不在 classpath 内），Spring 默认找不到。`addResourceHandler("/uploads/**")` 告诉 Spring：当浏览器请求 `http://localhost:8080/uploads/avatar/xxx.jpg` 时，去文件系统的 `D:\project\uploads\avatar\` 目录下找文件并返回。

**完整图片访问链路：**
```
用户上传头像：POST /api/user/avatar { file }
```

### 3.4 `MybatisPlusConfig.java` — 分页插件

**文件路径：** `backend/src/main/java/com/campus/config/MybatisPlusConfig.java`

```java
@Configuration
public class MybatisPlusConfig {

    @Bean
    public MybatisPlusInterceptor mybatisPlusInterceptor() {
        MybatisPlusInterceptor interceptor = new MybatisPlusInterceptor();

        // 注册分页拦截器，指定 MySQL 数据库方言
        interceptor.addInnerInterceptor(new PaginationInnerInterceptor(DbType.MYSQL));

        return interceptor;
    }
}
```

**工作原理（AOP 拦截）：**

MyBatis-Plus 的拦截器机制通过 JDK 动态代理拦截 Mapper 方法的 SQL 执行。当 Service 调用 `mapper.selectPage(page, wrapper)` 时：

```
① 用户代码：productMapper.selectPage(new Page<>(1, 12), wrapper)
         │
② MyBatis-Plus 代理拦截
         │
③ PaginationInnerInterceptor 介入
    ├─ 第一步：执行 COUNT 查询
    │   SQL: SELECT COUNT(*) FROM product WHERE status = 'ON_SALE'
    │   → 获取总数 total = 100
    │
    ├─ 第二步：拼接 LIMIT 子句
    │   原始 SQL: SELECT * FROM product WHERE status = 'ON_SALE'
    │   拼接后: SELECT * FROM product WHERE status = 'ON_SALE' LIMIT 0, 12
    │   → 获取第 1 页的 12 条记录
    │
    └─ 第三步：组装 Page 对象
        page.setRecords(数据列表)
        page.setTotal(100)
        page.setCurrent(1)
        page.setSize(12)
```

### 3.6 `RedisConfig.java` — Redis 序列化配置

**文件路径：** `backend/src/main/java/com/campus/config/RedisConfig.java`

```java
@Configuration
public class RedisConfig {

    @Bean
    public RedisTemplate<String, Object> redisTemplate(RedisConnectionFactory factory) {
        RedisTemplate<String, Object> template = new RedisTemplate<>();
        template.setConnectionFactory(factory);

        // Key 用 String 序列化 → 在 Redis 中可读
        template.setKeySerializer(new StringRedisSerializer());

        // Value 用 JSON 序列化 → 支持复杂对象，人可读
        template.setValueSerializer(new GenericJackson2JsonRedisSerializer());

        // Hash 结构的 Key 和 Value 同理
        template.setHashKeySerializer(new StringRedisSerializer());
        template.setHashValueSerializer(new GenericJackson2JsonRedisSerializer());

        return template;
    }
}
```

**为什么必须自定义序列化器？**

| | 默认 JDK 序列化 | 自定义 JSON 序列化 |
|------|------|------|
| Redis 中存储的样子 | 二进制乱码 | 可读 JSON 字符串 |
| 跨语言支持 | 仅 Java | 任意语言可读 |
| 额外空间开销 | 较小 | 多存 `@class` 字段，约 +30% |
| 反序列化前提 | 必须同 JVM 版本 | 只需知道 JSON 结构 |

Spring Boot 默认使用 `JdkSerializationRedisSerializer`，会把 Java 对象序列化为二进制字节流。这意味着在 Redis CLI 中 `GET key` 看到的是乱码，且 Redis 中的数据不能被其他语言的程序读取（如 Node.js、Python）。换成 `GenericJackson2JsonRedisSerializer` 后，Redis 中直接存 JSON，方便调试和跨语言集成。

### 设计模式——六张表的 Entity 使用完全一致的注解模式

```java
@Data                                        // Lombok: 编译时生成 getter/setter/toString/equals/hashCode
@TableName("`user`")                         // 映射数据库表名。user 是 MySQL 保留字，必须用反引号
public class User {

    @TableId(type = IdType.AUTO)             // 主键自增策略（对应数据库 AUTO_INCREMENT）
    private Long id;

    // 以下字段与数据库列名一致
    // 依靠 mybatis-plus.map-underscore-to-camel-case=true 自动转换
    // 数据库列 login_fail_count → Java 属性 loginFailCount（自动映射，无需 @TableField 指定）
    private Integer loginFailCount;

    @TableField(fill = FieldFill.INSERT)     // 标记：插入时自动填充值
    private LocalDateTime createdAt;         // 配合 MetaObjectHandler 使用

    @TableField(fill = FieldFill.INSERT_UPDATE)  // 标记：插入和更新时自动填充值
    private LocalDateTime updatedAt;         // 配合 MetaObjectHandler 使用
}
```

### 当前代码的一个已知问题

`@TableField(fill = FieldFill.INSERT)` 和 `FieldFill.INSERT_UPDATE` 标记了自动填充策略，但项目中**没有**实现 `MetaObjectHandler` 类。

没有 MetaObjectHandler 的话 MyBatis-Plus 不会自动填充这些字段。

**实际的填充逻辑在哪里？** 在数据库层面：

```sql
`created_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
`updated_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
```

数据库的 `DEFAULT CURRENT_TIMESTAMP` 确保了即使 Java 代码不传值，MySQL 也会自动填入当前时间。

**这个设计带来的隐性问题：**

```java
// Service 层代码
User user = new User();
user.setUsername("test");
user.setPassword("hashed");
userMapper.insert(user);     // 插入成功

// 问题：此时的 user.createdAt 可能为 null！
// 因为 MyBatis-Plus 默认不会回填自增字段和数据库默认值
// 要获取 createdAt，需要：
user = userMapper.selectById(user.getId());  // 重新查询
```

### 与数据库字段名不一致的特殊处理

**User 表：** `@TableName("\`user\`")` —— 用反引号包裹（MySQL 保留字）
**Order 表：** `@TableName("\`order\`")` —— 同上

其他四个表（product、message、review、report）不是保留字，直接写表名。

### 7.3 `GlobalExceptionHandler` — 全局异常处理

```java
@Slf4j
@RestControllerAdvice     // = @ControllerAdvice + @ResponseBody（返回 JSON 而非页面）
public class GlobalExceptionHandler {

    // 1. 参数校验异常（@Valid 失败时 MethodArgumentNotValidException）
    @ExceptionHandler(MethodArgumentNotValidException.class)
    @ResponseStatus(HttpStatus.BAD_REQUEST)  // 设置 HTTP 响应状态码为 400
    public Result<Void> handleValidation(MethodArgumentNotValidException e) {
        // 收集所有字段的错误消息，用逗号拼接
        // 例："用户名不能为空, 邮箱格式不正确"
        String msg = e.getBindingResult().getFieldErrors()
                .stream()
                .map(FieldError::getDefaultMessage)
                .collect(Collectors.joining(", "));
        return Result.fail(msg);
    }

    // 2. 认证异常（Spring Security 抛出的）
    @ExceptionHandler(AuthenticationException.class)
    @ResponseStatus(HttpStatus.UNAUTHORIZED)   // HTTP 401
    public Result<Void> handleAuth(AuthenticationException e) {
        return Result.fail(401, "认证失败: " + e.getMessage());
    }

    // 3. 权限不足异常
    @ExceptionHandler(AccessDeniedException.class)
    @ResponseStatus(HttpStatus.FORBIDDEN)      // HTTP 403
    public Result<Void> handleAccessDenied(AccessDeniedException e) {
        return Result.fail(403, "权限不足");
    }

    // 4. 业务参数异常（Service 中手动 throw new IllegalArgumentException("xxx")）
    @ExceptionHandler(IllegalArgumentException.class)
    @ResponseStatus(HttpStatus.BAD_REQUEST)     // HTTP 400
    public Result<Void> handleIllegalArgument(IllegalArgumentException e) {
        return Result.fail(e.getMessage());     // 直接返回 Exception 的消息文本
    }

    // 5. 兜底异常（所有未捕获的异常）
    @ExceptionHandler(Exception.class)
    @ResponseStatus(HttpStatus.INTERNAL_SERVER_ERROR)  // HTTP 500
    public Result<Void> handleOther(Exception e) {
        log.error("未捕获异常", e);            // 记录完整堆栈到日志
        return Result.fail(500, "服务器内部错误"); // 不向客户端泄露堆栈细节
    }
}
```

**异常处理链的匹配顺序：**

```
Service 中 throw new IllegalArgumentException("商品不存在")
  → Spring 异常解析器遍历 @ExceptionHandler
  → ① 不是 MethodArgumentNotValidException → 跳过
  → ② 不是 AuthenticationException → 跳过
  → ③ 不是 AccessDeniedException → 跳过
  → ④ 是 IllegalArgumentException → 匹配！
  → 执行 handleIllegalArgument()
  → 返回 Result(400, "商品不存在", null)
  → 前端拦截器捕获 code=400 → ElMessage.error("商品不存在")
```

**为什么需要兜底的 `Exception.class` 处理器？** 如果不加，未预料的异常（如 NullPointerException）会返回 Spring Boot 默认的 `500 Internal Server Error` HTML 错误页面，而不是 JSON。那样前端 Axios 会解析失败，用户体验极差。

### 8.2 `JwtTokenProvider.java` — Token 生成与验证

```java
@Component
public class JwtTokenProvider {

    private final SecretKey key;        // HMAC-SHA256 签名密钥
    private final long expirationMs;    // Token 有效期（毫秒）

    // 构造函数注入（通过 @Value 从配置文件读取）
    public JwtTokenProvider(
            @Value("${app.jwt.secret}") String secret,      // Base64 编码的密钥
            @Value("${app.jwt.expiration}") long expirationMs) {

        // ① 解码 Base64 字符串为字节数组
        byte[] keyBytes = Base64.getDecoder().decode(secret);

        // ② 用字节数组构建 HMAC-SHA256 密钥对象
        //    Keys.hmacShaKeyFor() 会自动验证密钥长度 >= 256 bits
        this.key = Keys.hmacShaKeyFor(keyBytes);
        this.expirationMs = expirationMs;
    }

    // === 生成 JWT Token ===
    public String generateToken(Long userId, String role) {
        Date now = new Date();
        return Jwts.builder()              // Builder 模式
                .subject(String.valueOf(userId))  // sub: "3"
                .claim("role", role)              // 自定义声明: role = "USER"
                .issuedAt(now)                    // iat: 签发时间
                .expiration(new Date(now.getTime() + expirationMs))  // exp
                .signWith(key)                    // HMAC-SHA256 签名
                .compact();                       // 生成最终字符串
    }

    // === 从 Token 中提取 userId ===
    public Long getUserId(String token) {
        return Long.parseLong(parseClaims(token).getSubject());
    }

    // === 从 Token 中提取 role ===
    public String getRole(String token) {
        return parseClaims(token).get("role", String.class);
    }

    // === 验证 Token 有效性 ===
    public boolean validate(String token) {
        try {
            parseClaims(token);     // 解析 + 验签 + 检查过期
            return true;            // 全部通过 → 有效
        } catch (JwtException | IllegalArgumentException e) {
            return false;           // 签名错误 / 过期 / 格式错误
        }
    }

    // === 私有方法：解析 JWT Claims ===
    private Claims parseClaims(String token) {
        return Jwts.parser()           // 创建解析器
                .verifyWith(key)       // 设置验证密钥
                .build()               // 构建（线程安全，可重用）
                .parseSignedClaims(token) // 解析并验签
                .getPayload();         // 提取 Payload（Claims 对象）
    }
}
```

### 8.3 `JwtTokenFilter.java` — 请求过滤器

```java
@Component
@RequiredArgsConstructor
public class JwtTokenFilter extends OncePerRequestFilter {
    // OncePerRequestFilter：保证每个 HTTP 请求只经过一次此过滤器
    // 即使请求在内部 forward 多次，也不会重复过滤

    private final JwtTokenProvider jwtTokenProvider;

    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                    HttpServletResponse response,
                                    FilterChain filterChain)
            throws ServletException, IOException {

        // 步骤 1：提取 Token
        String token = extractToken(request);

        // 步骤 2：如果 Token 存在且有效，设置认证信息
        if (StringUtils.hasText(token) && jwtTokenProvider.validate(token)) {
            Long userId = jwtTokenProvider.getUserId(token);
            String role = jwtTokenProvider.getRole(token);

            // 步骤 3：构建 Spring Security 的 Authentication 对象
            UsernamePasswordAuthenticationToken authentication =
                    new UsernamePasswordAuthenticationToken(
                            userId,           // principal
                            null,             // credentials（JWT 模式不需要）
                            List.of(new SimpleGrantedAuthority("ROLE_" + role))
                    );

            // 步骤 4：存入 SecurityContext
            SecurityContextHolder.getContext().setAuthentication(authentication);
        }

        // 步骤 5：继续过滤器链（无论是否认证都放行）
        filterChain.doFilter(request, response);
    }

    // === 从 HTTP Header 中提取 Bearer Token ===
    private String extractToken(HttpServletRequest request) {
        String header = request.getHeader("Authorization");
        if (StringUtils.hasText(header) && header.startsWith("Bearer ")) {
            return header.substring(7);   // 跳过 "Bearer " 7 个字符
        }
        return null;
    }
}
```

**为什么 Token 无效也继续执行（不直接返回 401）？**
- 不是所有请求都需要认证（比如公开的商品列表）
- 如果直接返回 401，公开接口也无法访问
- 正确的做法：过滤器只负责"如果 Token 有效就设置认证"，授权检查由 Spring Security 的 `authorizeHttpRequests` 配置在后续处理

**`SimpleGrantedAuthority("ROLE_" + role)` 中的 `ROLE_` 前缀：**
- 这是 Spring Security 的约定：角色自动加 `ROLE_` 前缀
- `user.getRole() = "ADMIN"` → 最终权限为 `ROLE_ADMIN`
- `SecurityConfig` 中 `hasRole("ADMIN")` 实际匹配的是 `ROLE_ADMIN`
- 如果用 `hasAuthority("ROLE_ADMIN")` 效果相同（但 hasRole 自动加前缀）

```java
// 在 Service 层定义状态转换映射
private static final Map<OrderStatus, Set<OrderStatus>> ALLOWED_TRANSITIONS = Map.of(
    PENDING_PAYMENT,  Set.of(PENDING_SHIPMENT, CANCELLED),
    PENDING_SHIPMENT, Set.of(PENDING_RECEIPT),
    PENDING_RECEIPT,  Set.of(COMPLETED)
);

private void validateTransition(Order order, OrderStatus target) {
    Set<OrderStatus> allowed = ALLOWED_TRANSITIONS.get(order.getStatus());
    if (allowed == null || !allowed.contains(target)) {
        throw new IllegalArgumentException(
            "不能从 " + order.getStatus() + " 转为 " + target);
    }
}
```

### 12.3 `stores/user.js` — Pinia 状态管理

```js
export const useUserStore = defineStore('user', () => {
    // === 状态（响应式 + 持久化初始化） ===
    const token = ref(localStorage.getItem('token') || '')
    const user = ref(JSON.parse(localStorage.getItem('user') || 'null'))

    // === 计算属性 ===
    const isLoggedIn = computed(() => !!token.value)
    const isAdmin = computed(() => user.value?.role === 'ADMIN')

    // === 操作 ===
    function setAuth(t, u) {
        token.value = t                    // 更新 Pinia（响应式，所有组件立即感知）
        user.value = u
        localStorage.setItem('token', t)   // 持久化（刷新后恢复）
        localStorage.setItem('user', JSON.stringify(u))
    }

    function logout() {
        token.value = ''
        user.value = null
        localStorage.removeItem('token')
        localStorage.removeItem('user')
    }

    return { token, user, isLoggedIn, isAdmin, setAuth, logout }
})
```

**双层存储机制：**

```
┌────────────────────────────────────┐
│         Pinia Store (内存)          │  ← 响应式，组件间共享
│         token / user               │  ← 页面刷新就丢失
└────────────────────────────────────┘
               ↕ 同步
┌────────────────────────────────────┐
│     localStorage (浏览器磁盘)        │  ← 持久化，刷新后仍存在
│     'token' / 'user'               │  ← 不响应式
└────────────────────────────────────┘
```

### 12.4 `api/index.js` — Axios 封装

```js
const api = axios.create({
    baseURL: '/api',    // 所有请求自动加 /api 前缀
    timeout: 15000,     // 15 秒超时
})

// ① 请求拦截器：自动注入 JWT
api.interceptors.request.use((config) => {
    const userStore = useUserStore()
    if (userStore.token) {
        config.headers.Authorization = `Bearer ${userStore.token}`
    }
    return config
})

// ② 响应拦截器：统一错误处理 + 自动解包
api.interceptors.response.use(
    (res) => {
        const data = res.data           // Spring Boot 返回的 JSON
        if (data.code !== 200) {
            ElMessage.error(data.message)   // 业务错误 → 弹窗提示
            return Promise.reject(new Error(data.message))
        }
        return data     // 组件中直接拿到 { code:200, data: ..., message: "success" }
    },
    (err) => {
        if (err.response?.status === 401) {   // HTTP 401 → 强制登出
            const useStore = useUserStore()
            useStore.logout()
            window.location.href = '/login'
        }
        ElMessage.error(err.response?.data?.message || '网络错误')
        return Promise.reject(err)
    }
)
```

**Axios 调用链路：**

```
Vue 组件: await productApi.list({ keyword: '手机', page: 1, size: 12 })
  │
  ├─ productApi.list(params) → api.get('/products', { params })
  │    其中 params 被 Axios 序列化为 URL query string: /api/products?keyword=手机&page=1&size=12
  │
  ├─ 请求拦截器：读取 Pinia → 设置 Authorization header
  │
  ├─ 发送 HTTP 请求
  │   由于 Vite 代理：localhost:5173/api/products → localhost:8080/api/products
  │
  ├─ 后端返回 JSON
  │   { code: 200, message: "success", data: { records: [...], total: 100, page: 1, size: 12 } }
  │
  ├─ 响应拦截器：
  │   code === 200 → 返回整个 res.data 给组件
  │   code !== 200 → ElMessage.error + reject
  │
  └─ 组件中：const res = await productApi.list(...)
      res.data.records  → 商品列表
      res.data.total    → 总数
```

**Vite 代理原理：**


## 十四、已知问题与重构方向

### 严重问题

| # | 问题 | 位置 | 影响 | 修复方案 |
|------|------|------|------|------|
| 1 | 缺少 `MetaObjectHandler` | 所有 Entity 的 createdAt/updatedAt | insert/update 后拿不到自动填充的时间 | 新增 `MyMetaObjectHandler` 类实现 `MetaObjectHandler` 接口 |
| 2 | 注册不捕获 DuplicateKeyException | AuthService.register() | 并发注册时抛 500 非友好错误 | `catch(DataIntegrityViolationException e)` 返回 "用户名/邮箱/手机号已存在" |
| 3 | ProductController.create 参数太碎 | ProductController | 7 个 @RequestParam，不适合 JSON 客户端 | 改为 `@RequestBody CreateProductRequest` + 单独的图片上传端点 |
| 4 | admin 密码硬编码 | init.sql 第 131 行 | BCrypt hash 可能不匹配 | 改为应用启动时用 Java 代码自动创建 admin |
| 5 | `OrderService` 全桩 | OrderService.java | 订单下单/支付/发货/收货全部不可用 | C 实现状态机逻辑 |

### 中等问题

| # | 问题 | 位置 | 影响 | 修复方案 |
|------|------|------|------|------|
| 6 | `@EnableMethodSecurity` 只有一处使用 | AdminController | 配置过度 | 统一策略：要么 URL 级，要么方法级 |

### 设计讨论

| 话题 | 当前设计 | 替代方案 | 取舍 |
|------|------|------|------|
| Entity status 用 String 而非 Enum | 类型不安全但灵活 | 用 Enum + TypeHandler | 课程项目选简单方案合理 |
| 软删除用 status 字段 | 需手动在查询中排除 DELETED | 用 MyBatis-Plus @TableLogic | 当前更直观，但代码量多 |
| Service 返回 Result 对象 | 耦合了 Web 层概念 | Service 返回 POJO，Controller 包装 Result | 当前更简洁—Service 可以直接返回错误 |
| Controller 中每个类重复 getUserId() | 代码重复 | 抽取 BaseController 或 AOP | 当前每个类才3行，抽取收益不大 |
